# 🏦 05_pjt - 금융 자산 정보 토론 게시판 2

## 🎯 프로젝트 개요
본 프로젝트는 기존의 금융 자산 커뮤니티 서비스에 **사용자 인증(Authentication) 및 프로필 기능**을 확장하고, 게시글에 대한 **권한 관리(Authorization)**를 강화하는 것을 목표로 진행되었습니다. 
Django의 기본 유저 모델을 대체하는 커스텀 유저 모델을 구현하여 서비스에 특화된 사용자 정보를 관리하고, MVT 패턴을 활용하여 안정적인 웹 서비스를 구축했습니다.

---

## 🛠 기술 스택
- **Language**: Python 3.11
- **Framework**: Django 5.2
- **Database**: SQLite3
- **Environment**: python-dotenv, venv
- **Version Control**: Git, GitLab

---

## 💡 구현 기능 설명

### 1. 사용자 관리 (Accounts App)
- **커스텀 유저 모델 (F501)**: `AbstractUser`를 상속받아 `nickname`(필수), `interest_stocks`(관심종목), `profile_image`(프로필 사진) 필드 추가
- **회원가입 및 자동 로그인 (F502)**: 회원가입 성공 시 세션을 발급하여 자동으로 로그인 상태 전환
- **로그인/로그아웃 (F503, F504)**: 인증 상태에 따른 네비게이션 바(Header) UI 동적 렌더링
- **비밀번호 변경 (F505)**: 현재 비밀번호 확인 및 불일치 시 유효성 검사 에러 메시지 출력
- **프로필 페이지 (F507)**: 유저의 프로필 이미지, 닉네임, 관심 종목 리스트 및 해당 유저가 작성한 게시글 목록을 필터링하여 제공

### 2. 커뮤니티 관리 (Community App)
- **게시글 권한 제어 (F506)**: 
  - `@login_required` 데코레이터를 활용하여 로그인한 사용자만 글 작성 가능
  - 게시글 작성 시 `Post.author` 필드에 현재 로그인한 유저의 `username` 자동 저장
  - 게시글 수정 및 삭제는 작성자 본인에게만 버튼이 노출되며, 백엔드 로직에서도 권한 검증 수행

## 🧠 분석 흐름 및 학습 내용

### 1. 앱(App)의 분리와 역할 분담
- 처음에는 모든 기능을 하나의 앱에서 처리하려 했으나, 장고의 설계 철학에 따라 기능별로 앱을 분리(`community`, `accounts`)하는 것의 중요성을 배웠습니다.
- 인증(Authentication)과 관련된 모든 기능은 `accounts` 앱으로 독립시켜 코드의 유지보수성과 재사용성을 높였습니다.

### 2. 커스텀 유저 모델 적용 시점
- 프로젝트 중간에 유저 모델을 변경하면 DB 마이그레이션이 심각하게 꼬일 수 있다는 것을 배웠습니다. `AUTH_USER_MODEL` 설정은 프로젝트 초기 세팅 단계에서 가장 먼저 수행해야 함을 명확히 인지했습니다.

### 3. 템플릿 상속과 UI 제어
- `base.html`을 활용한 템플릿 상속(`{% extends %}`)을 통해 중복 코드를 최소화했습니다. 
- `user.is_authenticated` 템플릿 태그를 활용해 사용자의 로그인 상태에 따라 헤더의 버튼(로그인/회원가입 vs 프로필/로그아웃)이 다르게 보이도록 UX를 개선했습니다.

## 🚨 어려웠던 점 및 해결 과정 (Troubleshooting)

오늘 프로젝트를 진행하며 수많은 에러와 마주했지만, 하나씩 원인을 분석하고 해결해 나갔습니다.

1. **의존성 패키지 누락 에러 (`ModuleNotFoundError`, `Pillow`)**
   - **문제**: `settings.py`에서 환경변수를 읽어오지 못해 서버가 터지거나, 커스텀 유저 모델에서 `ImageField`를 사용할 때 마이그레이션 에러가 발생했습니다.
   - **해결**: `pip install python-dotenv` 및 `pip install Pillow`를 통해 필요한 라이브러리를 가상환경에 설치하여 해결했습니다. 명세서의 준비사항을 꼼꼼히 읽는 것의 중요성을 깨달았습니다.

2. **가상환경(venv) 및 경로 설정 이슈**
   - **문제**: `manage.py`를 실행할 때 파일 경로를 찾지 못하거나, 가상환경이 `venv`와 `.venv` 두 개로 나뉘어 충돌하는 문제가 있었습니다.
   - **해결**: CLI 환경에서 현재 디렉토리 위치(`pwd`, `ls`)를 파악하여 `skeleton` 폴더 내부로 정확히 진입한 뒤 명령어를 실행했습니다. 불필요한 `.venv`는 삭제하고 하나의 가상환경으로 통일하여 패키지를 관리했습니다.

3. **앱 중복 등록 에러 (`ImproperlyConfigured`)**
   - **문제**: `settings.py`의 `INSTALLED_APPS`에 `community` 앱이 중복으로 들어가 있어 마이그레이션이 거부되었습니다.
   - **해결**: 코드를 찬찬히 뜯어보며 중복된 요소를 제거하여 장고가 앱 라벨을 정상적으로 식별하도록 고쳤습니다.

4. **템플릿 UI 중복 렌더링 문제 (로고가 2개 뜨는 현상)**
   - **문제**: `base.html`을 수정하는 과정에서 `<header>` 태그가 부모와 자식 템플릿 모두에 들어가거나 중복 작성되어 화면 상단이 깨졌습니다.
   - **해결**: HTML 구조를 다시 설계하여 뼈대가 되는 `base.html`에만 네비게이션 바를 단일화시키고, `{% block content %}`를 통해 하위 페이지들이 알맞게 끼워지도록 정리했습니다.

## 🏃‍♂️ 실행 방법

```bash
# 1. 가상환경 생성 및 활성화
python -m venv venv
source venv/Scripts/activate

# 2. 패키지 설치
pip install -r requirements.txt
pip install Pillow python-dotenv

# 3. 데이터베이스 마이그레이션
python manage.py makemigrations accounts
python manage.py migrate

# 4. 서버 실행
python manage.py runserver

## 💡 느낀 점 (회고)

이번 관통 프로젝트를 진행하면서 가장 크게 느낀 점은 **"초기 환경 설정과 프레임워크 구조 이해의 중요성"**입니다. 
단순히 명세서에 적힌 코드를 타이핑하는 것을 넘어, 아래와 같은 과정들을 통해 백엔드 개발자로서 한 층 성장할 수 있었습니다.

### 1. 장고(Django)의 설계 철학 체감
기존 커뮤니티 게시판 기능에 덧붙여 `accounts` 앱을 직접 생성하고 분리해 보면서, 장고가 왜 기능별로 앱(App)을 모듈화하도록 권장하는지 그 이유를 명확히 이해했습니다. 인증(Authentication) 로직을 분리함으로써 코드의 가독성과 유지보수성이 크게 향상됨을 느꼈습니다.

### 2. 에러를 두려워하지 않는 디버깅 역량 향상
가상환경(`venv`) 세팅부터 패키지 누락(`Pillow`, `python-dotenv`), 앱 중복 등록 에러(`ImproperlyConfigured`), 그리고 Git 업로드 꼬임 현상까지 수많은 에러와 마주했습니다. 처음에는 터미널에 쏟아지는 붉은색 Traceback 메시지가 두려웠지만, 에러 로그를 끝까지 읽고 원인을 역추적하여 하나씩 해결해 나가는 과정에서 큰 성취감을 느꼈습니다. 이제는 에러가 발생해도 "어디서부터 꼬였는지 찾아보자"는 논리적인 접근이 가능해졌습니다.

### 3. 버전 관리(Git)와 협업의 기본기
코드가 꼬였을 때 복구하는 방법, 그리고 `.gitignore`를 활용해 수천 개의 불필요한 가상환경 파일이 서버에 올라가는 것을 방지하는 방법을 확실히 익혔습니다. 이는 앞으로 진행될 팀 프로젝트와 실무 협업에서 아주 중요한 밑거름이 될 것이라 확신합니다.

결과적으로 이번 프로젝트는 겉으로 보이는 UI를 넘어, 서비스의 근간이 되는 **유저 모델 변경, 데이터베이스 마이그레이션, 권한(Authorization) 제어**라는 보이지 않는 백엔드의 핵심 흐름을 제 손으로 뚫어낸 뜻깊은 경험이었습니다.
